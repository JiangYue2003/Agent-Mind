# 欧米伽科技企业信息安全与数据合规管理规范指引 (V 5.0.2)

## 1. 总则与安全基线准则

### 1.1 制定目的与效力
鉴于企业业务涉及高并发架构与海量用户数据，为构筑坚实的安全底座，防范数据泄露、非法提权及核心资产流失，特发布本规范。本文件为企业“红头文件”级别，具有最高内部技术合规效力，覆盖所有研发、运维、测试及数据分析人员。

### 1.2 零信任网络访问 (Zero Trust Access)
企业内部网络全面实行“零信任”架构：
* **物理与网络边界：** 任何对生产环境（Production）、预发环境（Staging）及核心代码仓库的访问，必须通过企业级 VPN 接入，并强制开启多因素身份验证（MFA）。
* **最小权限原则 (PoLP)：** 系统访问权限的开通需基于“当前岗位绝对必要”原则。跨部门调岗或离职时，系统必须在 2 小时内自动触发并完成权限回收（Revocation）。
* **临时提权 (JIT Access)：** 研发人员排查线上故障需申请临时高权限（如数据库只读权限）时，需通过审批流获取时效不超过 4 小时的临时 Token，且全程审计录屏。

## 2. 源代码安全与 AI 辅助开发规范

随着微服务架构的演进及智能化开发工具的普及，代码防泄露与知识产权保护成为核心焦点。

### 2.1 代码提交与审查规范
* **静态应用安全测试 (SAST)：** 所有合入主干（Main/Master）的代码，在 CI/CD 流水线中必须强制通过 SonarQube 等代码扫描工具的安全检查。阻断级（Blocker）及严重级（Critical）漏洞必须清零方可合并。
* **开源组件引入：** 严禁私自引入未经验证的第三方开源依赖。所有引入的包（如 Go Modules）必须经过安全团队的依赖性漏洞（SCA）扫描，且许可证（License）不得为强传染性开源协议（如 GPL）。

### 2.2 AI 辅助编程与大模型使用红线
为拥抱 AI 提升研发效能同时管控风险，特制定以下红线：
1.  **公有云 AI 模型禁令：** 严禁将包含公司业务逻辑、未公开算法、内部系统 IP 及数据库表结构的源代码，直接复制粘贴至未经企业级隐私协议授权的公开大语言模型（如公共版 ChatGPT、公共版 Claude 等）对话框中。
2.  **合规 AI 工具清单：** 员工仅可使用公司统一采购并配置了数据隔离协议的内部 AI 服务（如企业版 GitHub Copilot、本地化部署的 AI 编码插件，或已接入企业内网网关的受控模型 API）。
3.  **MCP（模型上下文协议）限制：** 在配置本地 AI 编码环境时，严禁将包含生产环境配置的 `.env` 文件或存放核心鉴权逻辑的目录加入 AI 的上下文读取白名单。

## 3. 凭证与 API 密钥生命周期管理

硬编码秘钥是系统安全的高危风险点，必须执行严格的生命周期闭环管理。

### 3.1 秘钥存储与调用
* **硬编码零容忍：** 严禁在源代码文件（包括但不限于 `.go`, `.java`, `.py` 等）、配置文件（如 `yaml`, `json`）或前端代码中明文硬编码（Hardcode）任何敏感凭证。敏感凭证包括：数据库密码、云厂商 AK/SK、第三方支付网关私钥、微服务间 gRPC 通信证书等。
* **统一凭证中心：** 所有微服务启动时，必须通过动态注入或请求企业凭证配置中心（如 HashiCorp Vault 或加密的 Etcd 集群）获取临时凭证。

### 3.2 秘钥轮转机制 (Rotation)
不同层级的凭证需执行差异化的强制轮转周期：
| 凭证级别        | 适用场景                                 | 强制轮转周期    | 告警触发阈值                 |
| :-------------- | :--------------------------------------- | :-------------- | :--------------------------- |
| **L1 核心凭证** | 生产库 Root 密码、核心支付网关私钥       | 每 30 个自然日  | 到期前 7 天触发 P1 级别告警  |
| **L2 内部凭证** | 内部微服务 RPC 相互调用 Token            | 每 90 个自然日  | 到期前 15 天触发 P2 级别告警 |
| **L3 外部调用** | 非核心第三方接口 API Key（如短信、物流） | 每 180 个自然日 | 到期前 30 天触发普通工单提醒 |

## 4. 数据合规与微服务安全通信

### 4.1 敏感数据分级与脱敏 (Data Masking)
针对个人身份信息（PII，如身份证号、手机号、银行卡号）：
* **持久化层：** 写入核心数据库（如 MySQL/PostgreSQL）时必须采用 AES-256 及以上强度算法进行字段级加密。
* **缓存层：** 写入分布式缓存（如 Redis）的 PII 数据必须经过哈希或不可逆掩码处理，严禁在缓存中存储 PII 明文。
* **日志层：** 应用日志（如通过 ELK 收集的日志）在输出前必须经过日志清洗脱敏中间件拦截，手机号仅可保留前 3 后 4 位。

### 4.2 微服务间通信加密
* 所有跨机房或跨可用区的服务间调用（含 HTTP 及 gRPC 协议），必须开启 TLS 1.2 及以上版本的双向认证（mTLS），严禁明文传输业务数据。

## 5. 安全事件定级与应急响应 (SLA)

任何安全违规或疑似数据泄露事件，需遵循以下“先阻断，后上报”的响应机制：

### 5.1 事件定级标准
* **S0 级（致命事件）：** 生产数据库遭遇脱裤、核心业务系统被植入勒索病毒、公司主站证书私钥在 GitHub 公开仓库泄露。
* **S1 级（严重事件）：** 开发测试环境的高权限账号泄露、员工违规向外部 AI 投喂核心业务代码片段、内网发生大面积蠕虫感染。
* **S2 级（一般违规）：** 发现代码库中存在低危漏洞、非核心应用未按时轮转 L3 级外部 API Key。

### 5.2 响应时效要求
* **上报要求：** 发现者必须在 **15 分钟内** 通过内部安全告警群或专线电话直报安全响应中心（SRC）。严禁瞒报、漏报或私自掩盖（瞒报者一律按解除劳动合同处理，并视情况移交司法）。
* **处置要求：** 针对 S0 及 S1 级事件，安全团队需在 30 分钟内完成初步止血（如阻断 IP、封禁异常账号、吊销泄露凭证）。
